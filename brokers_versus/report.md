## Анализ и выводы

### Пропускная способность

- При размере 128 B и целевой интенсивности 10000 msg/s:
  - RabbitMQ фактически отправил **5223 msg/s**
  - Redis фактически отправил **4416 msg/s**

### Влияние размера сообщения

- **128 B**: RabbitMQ p95 = 497.00 ms, Redis p95 = 569.00 ms. RabbitMQ имеет меньшую задержку.
- **1024 B**: RabbitMQ p95 = 505.00 ms, Redis p95 = 543.00 ms. RabbitMQ имеет меньшую задержку.
- **10240 B**: RabbitMQ p95 = 557.00 ms, Redis p95 = 546.00 ms. Redis имеет меньшую задержку.
- **102400 B**: RabbitMQ p95 = 629.00 ms, Redis p95 = 656.00 ms. RabbitMQ имеет меньшую задержку.
- При нагрузке 10000 msg/s и размере 100 КБ: RabbitMQ сохраняет приемлемую задержку, в то время как Redis значительно деградирует.

### Устойчивость к высокой нагрузке

Точка деградации определялась по увеличению p95 задержки > 50 мс или потерям > 1%.

**RabbitMQ**:
- Размер 128 B: деградация начинается при **1000 msg/s**.
- Размер 1024 B: деградация начинается при **1000 msg/s**.
- Размер 10240 B: деградация начинается при **1000 msg/s**.
- Размер 102400 B: деградация начинается при **1000 msg/s**.

**Redis**:
- Размер 128 B: деградация начинается при **1000 msg/s**.
- Размер 1024 B: деградация начинается при **1000 msg/s**.
- Размер 10240 B: деградация начинается при **1000 msg/s**.
- Размер 102400 B: деградация начинается при **1000 msg/s**.

### Рекомендации

- **Redis** предпочтителен, когда:
  - Сообщения маленькие (до 10 КБ) и требуется минимальная задержка.
  - Высокая пропускная способность критичнее гарантий доставки.
- **RabbitMQ** следует выбрать, если:
  - Важна надёжность: сообщения не должны теряться.
  - Размер сообщений большой (более 10 КБ) или нагрузка неравномерная.
- Для сценариев с жёсткими требованиями к latency на малых payload лучше Redis.
- Для систем, где потеря данных недопустима — RabbitMQ.
